What is HTML?

The dawn of the web was defined by raw HTML. No fancy colors, no animations, no React + Express SPAs with externally hosted databases... just pure, unadulterated text, some text bigger than other text, which often linked to other text. Everything was text. Need an example?

And that's all that it is. Any page rendered with exclusively HTML will look just like that. Below this paragraph is what this page looks like with no CSS. Keep in mind that HTML was birthed in a time where the web was expected to be used mostly by scientists & researchers; TimBL (the creator of HTML and the World Wide Web) certainly didn't expect that HTML would be the foundation powering giant websites like Amazon in the future.

If you're reading this article, that's might all be stuff you don't really care about anyways. My point is that HTML is deceptively simple. It wasn't designed to be anything fancy, and it certainly isn't. I would say you can get completely comfortable with HTML in a day. Mastering it may take, like, a couple days, man.

The basics

HTML at a high-level is just a bunch of tags that describe the content they contain. Some have content in within two tags, others don't. Here's an example of the former:

<p>Pipertext fuckin' rules!</p>

Take note of the tags before and after the text, and also know that <p> stands for "paragraph". The first tag is the opening tag. It lets the browser know, "hey, a paragraph starts here." The second tag (and final tag for this particular paragraph element) tells the browser that the paragraph is now complete. Everything in-between these tags is the text that makes up the paragraph.

The second kind of tag is called a "void tag" to differentiate it from standard, non-void tags. A void tag only has one tag, the opening tag, and does not need to be closed. A fine example is the <img> tag:

<img src='/images/pipertextLogo.png' alt='The Pipertext logo'>

An image doesn't need a closing tag, because the image itself is referenced as an attribute within the <img> tag. But what is an attribute?

Attributes

You'll notice I actually have two attributes in the above <img> tag. One is src, which tells the browser where to find the desired image; obviously, "src" stands for "source". If I changed this attribute's value to something like "epicCookie.png" and there was a picture of an epic cookie at the file path specified, then it would show that instead.

The second attribute is alt which is short for "alternate text". Alternate text is used in two cases: when the image is unable to be retrieved from the server, and when someone is using a tool which reads it off to them automatically (which may be you..!). This is usually in the case of visually impaired individuals who may have a hard time seeing an image normally, so they depend on the alternate text being read out to them.

These two attributes demonstrate the use of attributes as a whole. They provide additional information regarding the element they are found in. Do note that, in the case of non-void tags, attributes are always found within the opening tag. Closing tags are always just that, closing tags.

Types of elements

There exists somewhere in the ballpark of 110 HTML elements overall. The reason there isn't a concrete number is that browsers choose to implement very new or very niche elements on their own accord, and many old, sometimes browser-specific elements such as <marquee> are still around despite being considered "deprecated" by all major browsers. The primary tags that you'll make us of frequently in your webpage-crafting career are universally supported.

I like to group HTML tags like the following:

  • Text-level elements, like for headings and paragraphs
  • Graphic elements, stuff like images and horizontal rules
  • Semantic elements, tags which define things like "this is the main section"
  • Metadata elements, which relay information related to the page, like associated files or keywords
  • Obsolete elements, sometimes fun to use but generally obscure/not practical

I feel that these categories mostly sum up what you have to work with. Text-level elements are like <p>, which defines a simple paragraph. Headings are like <h1>, <h2>, <h3>, and so forth. One of my favorites (which you may notice I am using liberally on this page) is <code>, which designates the contained text as being a code segment of some kind. Most browsers will apply a monospace font to text found within a <code> element.

Graphic elements are like the aforementioned <img> tag, or stuff like <hr> which simply defines a horizontal line. Semantic elements include not only stuff like <main> (used to indicate the main content of a page) and <time> (used to indicate that the contained text is depicting time), but also stuff like <button> which offer functionality as well as indicating what the element actually is. Metadata elements, which I have yet to discuss, are stuff like <link>. Here's an example of one of those for clarity:

<link href='/style.css' rel='stylesheet'>

This line of code tells the browser that there is a CSS stylesheet (which will add formatting, colors, layout changes, et cetera) related to this page, hence the rel attribute (short for "relation"), and where to find this stylesheet, hence the href attribute. Think of these as data about the webpage, rather than data for the webpage to use and display directly. I won't get into obsolete elements too much, but it's stuff like the aforementioned <marquee> tag, which is not an HTML standard and will work differently and sporadically across different browsers. Websites like Pipertext are commonly quite fond of these, as they do represent the fun, Wild West vibe of the oldweb, distinct from today's strict standards and lack of fun stuff like <marquee> and <blink>.

MDN and W3Schools both have extensive lists of the most commonly used HTML elements. I'd say give them a scroll and see what's out there!

What do I do with this wisdom?

Below is an example of some HTML in action. Study it deeply, young one...

<html> <head> <meta charset="UTF-8"> <meta name="viewport" content="width=device-width, initial-scale=1.0"> <title>pipertext</title> <link rel="icon" type="image/x-icon" href="/media/img/pipe.png"> <link rel="stylesheet" href="/style.css"> <script src="/scripts/typewriter.js" type="module" defer=""></script> </head> <body> <header> <h1>pipertext<h2> <h2>welcome to the embernet</h2> </header> <main> <p>Welcome to the first day of the rest of your life. Pipertext kicks your ass so hard the boot comes out the other end, flies to your mom's house, and gets dinner.</p> <h3>upcoming events</h3> <p>At <time datetime="2025-08-26T11:00:00-05:00">11:00 a.m. EST</time> tomorrow, something is gonna happen. I don't know. I can't come up with an example. Just imagine we're all going to take a siesta or something.</p> </main> <footer> <p>copyright pipertext 2025</p> </footer> </body> </html>

Now, you're probably saying, "holy shit dear webmaster, what is all of this magic and sorcery I have yet to be endowed with?" Do not fear, for at a second glance, it's all really quite self-explanatory.

Let's first take a look at the <head> tag. The head tag contains metadata about the page. Like I mentioned earlier, the webpage doesn't really display this data to the user (generally), but uses it to define what the webpage is or what resources it requires. In particular, take a look at the last two elements found within the <head> tag. One references the stylesheet, and the other references a JavaScript file. The user will never see this information unless they fat-finger Ctrl+Shift+I or something, but they will certainly notice if the stylesheet didn't load.

Now, to be fair, if you've looked at the middle two elements, the <title> element and first <link> element, you may have caught me in a lie... these two directly define data to be shown to the user. The difference is in the delivery method: let's take the <title> element for example; this information is not found on the webpage, opposed to to how something like <h1>Put that in your pipe and smoke it!</h1> would. Instead, it is always displayed in a way the browser dictates, almost always as text on the tab representing the page.

The top two <meta> tags are just that: metadata. Really, if I had to guess, at least half of web developers would be unable to tell you what these lines truly mean. Code editors like Visual Studio Code will generally add them in for you when generating boilerplate code. UTF-8 is basically just a character set, so that tag is telling the browser to use it. The second <meta> tag is honestly difficult to explain. Here's a Reddit thread where the purpose of the line was put into question. See, even I am part of that half! The smartest guy you know just showed you a shortcoming of his... just wow...

What practical purpose does this tool serve?

As I mentioned earlier, HTML wasn't intended to be such a widespread, necessary tool as it is today. But here we are, so learn to love it!

HTML acts as the foundation of any website. Without HTML, there is no webpage. The same can't be said for CSS or JavaScript, the other two pillars of the web. HTML doesn't really define layout, that's the job of CSS, but it does define structure. For example, this heading belongs in this header, or this footer. Maybe this paragraph belongs in the <main> tag, and that other one belongs in the <aside> tag. Good HTML practices basically allow your website to be read as though it's a newspaper when, for whatever bizarre reason, you don't have any CSS to help make it look pretty.

The essence of HTML

Getting familiar with HTML really will not take you long. Just reading this article probably is enough to get the gist; now when I tell you about the <content> element, you can probably have a vague idea of its function in your head.
